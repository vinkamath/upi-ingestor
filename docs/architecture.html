Architecture

How the pipeline moves money data from Gmail to Monarch, with Telegram in the loop.

Overview

UPI Ingestor is a personal finance automation pipeline built as a single Next.js application deployed on Vercel. It has no background workers or queues — the entire flow runs synchronously inside a single Vercel function invocation, triggered either by a daily cron job or a manual fetch from the dashboard.

Design principle: Every credential (Gmail OAuth tokens, Monarch password) is encrypted at rest using AES-256-GCM before being stored in Supabase. The ENCRYPTION_KEY never leaves your environment.

All database tables use Supabase Row Level Security — every policy is scoped to auth.uid() = user_id, so no user can read another user's transactions or rules, even via the JS client.

Pipeline Steps

1
Trigger

The Vercel cron fires GET /api/cron/poll at 3:30 UTC daily (configured in vercel.json). The UI can also trigger POST /api/gmail/fetch-now manually. Both call processUserTransactions(userId) in src/lib/pipeline.ts.

2
Gmail fetch — email-sources/gmail.ts

Loads the user's encrypted Gmail OAuth refresh token from gmail_connections, decrypts it, and calls the Gmail API to list messages matching the GMAIL_FETCH_LABEL (default: UPI) label received within the last GMAIL_FETCH_DAYS_BACK days. Message bodies are fetched individually and passed to the parser registry.

3
Parsing — parsers/hdfc.ts

Each email body is matched against the HDFC parser using regex patterns. The parser extracts amount, merchantRaw, bankRefId, occurredAt, and a normalized merchant key. Messages that don't match are flagged as parse errors and reported via Telegram if a bot is linked. The parser registry (parsers/index.ts) is intentionally additive — new bank parsers can be registered without changing the pipeline.

4
Dedup & persist

Each ParsedTransaction is checked against transactions by (user_id, bank_ref_id). Duplicates are skipped. New transactions are inserted with status: pending.

5
Categorization — categorizer/engine.ts

Two-pass lookup in priority order:

Pass 1 — Rules: user-defined rules stored in the rules table. Each rule has a match_type (regex / contains / equals), a target field (merchant / sender / body), and a pattern. Rules are evaluated in ascending priority order; first match wins.

Pass 2 — Merchant mappings: the normalized merchant key is looked up in merchant_mappings. Mappings are learned when a user responds to a Telegram prompt.

If neither pass produces a category, the transaction enters the human-in-the-loop flow.

6
Human-in-the-loop — Telegram Bot

Unknown transactions are marked needs_review. A Telegram inline keyboard is sent with the user's three most recently used categories plus a "Type new category" option. A row is inserted into pending_reviews linking the Telegram message to the transaction (7-day TTL).

When the user taps a button, POST /api/telegram/webhook receives a callback_query with callback_data: cat:<txId>:<category>. The handler saves a new merchant_mapping and triggers republish.

7
Publish — publishers/monarch/

The Monarch publisher decrypts the user's stored Monarch credential, calls their GraphQL createTransaction mutation, and updates the transaction row with status: published and the returned published_id. On failure, status: failed is set and the error is stored in raw_payload.publish_error for manual retry via POST /api/transactions/:id/republish.

Module Reference

Module Role Key exports
lib/pipeline.ts Pipeline orchestrator — runs fetch → parse → categorize → publish for one user processUserTransactions(userId)
lib/email-sources/gmail.ts Gmail API client; fetches and decodes messages matching the UPI label fetchGmailTransactions(userId)
lib/parsers/hdfc.ts Regex parser for HDFC UPI debit/credit notification emails parseHdfc(body)
lib/parsers/index.ts Parser registry — tries all registered parsers; returns first successful parse parseEmail(body)
lib/categorizer/engine.ts Two-pass categorization: rules → merchant mappings → unknown categorizeTransaction(supabase, userId, tx)
lib/categorizer/normalize.ts Merchant name normalization (lowercasing, punctuation stripping) normalizeMerchant(raw)
lib/merchant-mappings.ts CRUD helpers for the learned merchant→category mapping table saveMerchantMapping()
lib/publishers/monarch/ Monarch Money GraphQL publisher; handles credential decryption and mutation publisher.publish(userId, tx)
lib/telegram/client.ts Thin Telegram Bot API wrapper for sending messages with inline keyboards sendTelegramMessage(chatId, text, markup)
lib/crypto/encryption.ts AES-256-GCM encrypt/decrypt for credentials stored in Supabase encrypt(plain) · decrypt(enc)
lib/supabase/server.ts Server-side Supabase client (cookie-based session for App Router) createClient()
